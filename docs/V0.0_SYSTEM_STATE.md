## 2. Cryptographic Guarantees

### 2.1 RFC 6962 Merkle Compliance

The released implementation in `core/internal/forensics/merkle.go` uses Certificate Transparency style domain separation exactly as follows:

- Leaf hash: `SHA256(0x00 || leaf_bytes)`
- Internal node hash: `SHA256(0x01 || left_hash || right_hash)`
- Tree recursion: split at the largest power of two strictly less than `n`, then hash left and right subtrees recursively

This construction matters because it avoids the duplicate-last-leaf ambiguity that breaks non-RFC6962 trees. The release code does not pad, duplicate, or promote the final leaf by copying it. Inclusion proof elements are encoded as fixed 33-byte steps:

- byte `0x00`: sibling is on the left
- byte `0x01`: sibling is on the right
- remaining 32 bytes: sibling subtree hash

Proofs are generated bottom-up by `GenerateInclusionProof` and verified bottom-up by `VerifyInclusionProof`. The API export path (`core/internal/api/forensics_handler.go`) refuses to emit a bundle unless the proof it generated also verifies locally against the recomputed RFC6962 root. The offline verifier (`tools/verify_forensics.go`) repeats the same proof validation from the exported bundle and rejects on any mismatch.

### 2.2 Ed25519 Signing Paths

The released baseline contains four distinct Ed25519 trust applications.

1. WORM evidence signing
   - Signing key path: `/etc/ransomeye/worm_signing.key`
   - Implementation: `core/internal/crypto/worm.go`
   - Signed value: `SHA256(agent_id || 0x00 || event_id || 0x00 || event_type || 0x00 || logical_clock_be || 0x00 || payload)`
   - Persisted artifact: `worm_evidence.ed25519_sig`

2. Chained Merkle daily-root signing
   - Signing key path: `/etc/ransomeye/worm_signing.key`
   - Implementation: `core/internal/crypto/worm.go`, `core/internal/forensics/ledger_persist.go`, `core/internal/soc/control_worm.go`
   - Signed value: `SHA256(prev_root_hash || current_rfc6962_root)`
   - Persisted artifact: `merkle_daily_roots.ed25519_sig`

3. Forensic export bundle signing
   - Signing key path: `/etc/ransomeye/worm_signing.key`
   - Implementation: `core/internal/api/forensics_handler.go`
   - Signed value: `SHA256(canonical_json(bundle_without_bundle_signature))`
   - Persisted/exported artifact: `bundle_signature`

4. Signed common configuration
   - Signing key path: `/etc/ransomeye/pki/intermediate_ca.key`
   - Verification key source: Ed25519 public key extracted from `/etc/ransomeye/pki/intermediate_ca.crt`
   - Implementation: `installer/src/config_signer.rs`, `signed-config/src/lib.rs`
   - Signed value: `SHA256(canonical_json(common_config_without_integrity.signature))`
   - Persisted artifact: `/etc/ransomeye/config/common.yaml` with `integrity.signature`

The release therefore has no unsigned configuration trust path for database fingerprint policy or runtime client certificate paths. The installer normalizes the installed config, injects the PostgreSQL server fingerprint, forces `database.tls_enforced = true`, and signs the result before it is accepted by runtime code.

### 2.3 Canonical JSON Enforcement

Canonical JSON is not advisory in v0.0. It is a signature precondition.

`core/internal/forensics/canonical_json.go` enforces the following for forensic objects:

- Object keys are lexicographically sorted at every level
- Arrays preserve input order
- JSON is compact, with no insignificant whitespace
- Non-finite floating-point values are rejected
- Unsupported types are rejected

`signed-config/src/lib.rs` applies an even tighter canonical form for `common.yaml` by rebuilding the serialized object in a fixed field order after normalizing:

- `core.grpc_endpoint`
- `core.server_cert_fingerprint`
- `database.expected_server_fingerprint`
- `database.tls_enforced`
- `identity.node_id`
- `identity.role`
- `security.ca_cert_path`
- `security.client_cert_path`
- `security.client_key_path`

Any signature verification is therefore bound to deterministic bytes, not to YAML formatting, field ordering in source files, or parser-specific map iteration order.

### 2.4 Chained Merkle Root Logic

The release distinguishes between two cryptographic values:

- `merkle_daily_roots.merkle_root`: the RFC6962 root for the day’s ordered leaves
- `SHA256(prev_root_hash || merkle_root)`: the chained digest that is actually signed

`prev_root_hash` is empty for the first row in a tenant chain and then stores the prior RFC6962 daily root for every subsequent row. `core/internal/forensics/ledger_verify.go` verifies:

- row order by `computed_at ASC`
- genesis row has empty `prev_root_hash`
- each later row references the exact previous root
- each signature validates against the chained digest

This means daily roots are not isolated signatures. Each signature commits both to the current day’s tree and to chain continuity from the preceding root.

## 3. Network & Security Invariants

### 3.1 Loopback-Only Bindings

The v0.0 preservation baseline treats the following ports as local-only service surfaces:

- `127.0.0.1:50052` AI service
- `127.0.0.1:50053` SINE service
- `127.0.0.1:5432` PostgreSQL
- `127.0.0.1:6379` Redis
- `127.0.0.1:8443` Core HTTPS/WebSocket API

The released `scripts/full_system_validation.sh` requires all five ports to be present on `127.0.0.1` and rejects any of them on `0.0.0.0` or `::1`. `core/internal/compliance/registry.go` separately enforces that live internal listeners are loopback-only and that the monitored internal ports do not appear in `/proc/net/tcp6`.

Port `50051` is intentionally not part of the loopback-only set. It is the external agent/probe ingress surface and is protected by TLS 1.3 mutual authentication in the gRPC gateway.

### 3.2 TLS 1.3 Enforcement

TLS 1.3 is a released invariant in the following paths:

- Core gRPC ingress: `core/internal/gateway/server.go` builds a TLS config with `MinVersion = TLS1.3` and `ClientAuth = RequireAndVerifyClientCert`
- Core HTTPS API: `core/internal/soc/server.go` serves `:8443` with both minimum and maximum version pinned to TLS 1.3
- nginx front end: `installer/nginx/ransomeye.conf` sets `ssl_protocols TLSv1.3`
- PostgreSQL application transport: `core/internal/db/common.go` and `core/internal/db/tls_transport.go` require explicit TLS 1.3, fail if `InsecureSkipVerify` is true, and reject non-TLS1.3 connection states
- PostgreSQL server provisioning: `installer/src/postgres_tls.rs` writes `ssl_min_protocol_version = 'TLSv1.3'`

The release validation script uses the internal CA file `/opt/ransomeye/core/certs/ca.crt` and requires:

- successful TLS 1.3 handshake to `127.0.0.1:8443`
- explicit output line `Protocol: TLSv1.3`
- explicit output line `Verify return code: 0 (ok)`
- failed TLS 1.2 probe with a recognized protocol failure indicator

Any of those conditions missing is a release validation failure.

### 3.3 mTLS Architecture

The released mTLS trust model is not universal across every loopback path; it is scoped to specific transports and must be read exactly as implemented.

Enforced in v0.0:

- Agent and probe ingress on `:50051`
  - Server requires and verifies client certificates
  - Gateway extracts the Ed25519 public key from the client certificate
  - Envelope-declared `agent_ed25519_public_key` must match the certificate public key
  - Payload signature must verify before semantic parsing
  - Certificate CN must match `agent_id`

- PostgreSQL application channel on `127.0.0.1:5432`
  - Client presents certificate and key
  - Client trusts only the configured CA
  - Runtime verifies handshake completion, TLS 1.3, leaf presence, exact leaf fingerprint, exact loopback SAN, and loopback remote address
  - Database access is wrapped in `TrustedConn`, which panics if SQL is attempted before the transport trust gate passes

- nginx to Core API on loopback
  - Not client-authenticated
  - Does verify the upstream certificate against `/opt/ransomeye/core/certs/ca.crt`

Released asymmetry that must be preserved in the audit record:

- Historical Python sidecar code bound loopback gRPC on port 50052 with mutual TLS; that implementation is not present in this repo slice (use `RANSOMEYE_AI_ADDR` for an external peer when configured).
- `core/internal/ai/client.go` still dials that endpoint with `grpc.WithTransportCredentials(insecure.NewCredentials())`

The v0.0 baseline therefore contains a transport mismatch on the AI loopback path. That mismatch is real, frozen in the release lineage, and outside the checks performed by `scripts/full_system_validation.sh`.

### 3.4 No External Dependencies

The released system is designed to operate without outbound internet dependencies at install time or at runtime.

- `core/internal/compliance/registry.go` treats non-loopback resolvers in `resolv.conf` as an air-gap violation
- The same compliance gate rejects a default route in the route table
- Runtime validation is intentionally local-state based; it does not attempt outbound probes because outbound probing would itself violate the operating model
- nginx upstream trust uses a local CA bundle
- PostgreSQL, Redis, AI, SINE, and Core API dependencies are all local-host services in the single-node baseline
- Configuration trust, CA trust, and WORM signing all resolve from local files, not from remote KMS, SaaS PKI, or online attestation services

## 4. Forensic Guarantees

### 4.1 WORM Immutability Model

The v0.0 forensic model is split across filesystem immutability controls and database immutability controls.

Filesystem path:

- WORM payloads are sealed with AES-256-GCM in `core/internal/crypto/worm.go`
- The sealed blob is stored on disk and the release code checks for file mode `0444` after write in `core/internal/soc/control_worm.go`
- Export bundles include `sealed_blob_b64` as `nonce || ciphertext`, allowing signature verification without exposing the AES key

Database path:

- `migrations/006_worm_immutability.sql` installs `BEFORE UPDATE` and `BEFORE DELETE` triggers on `telemetry_events` and `worm_evidence`
- Trigger functions run as `SECURITY DEFINER` with `search_path = pg_catalog`
- `core/internal/db/validator/validation_runner.go` actively tests that update and delete attempts fail with immutability violations
- The same validator also checks immutable behavior on `governance_audit_log`, `incident_notes`, and `bundle_application_log`

The preserved release baseline is therefore append-oriented. Evidence creation is allowed; mutation and deletion of persisted forensic records are not.

### 4.2 Inclusion Proof Validation

The API export path does not trust stored proof material. It reconstructs proofs from ordered leaves at request time.

- `core/internal/api/forensics_handler.go` loads all `soc_merkle_leaf` values for the tenant and day in `merkle_position` order
- It locates the target leaf by byte equality
- It generates an RFC6962 proof from the ordered leaf set
- It verifies that proof locally against the stored daily root
- It recomputes the daily root from the full leaf set and rejects if the stored root differs

The exported bundle therefore contains:

- the evidence row
- the raw canonical leaf bytes as `soc_merkle_leaf_b64`
- the Merkle audit path as `merkle_proof`
- the stored `merkle_root`
- the prior-day linkage value `merkle_prev_root_hash`
- the signed root record signature
- the bundle signature

### 4.3 Export Bundle Verification

`tools/verify_forensics.go` is the released offline verifier. It is fail-closed and rejects the bundle unless all of the following hold:

- the bundle signature is present, decodes to 64 bytes, and verifies against the canonical bundle representation
- the canonical leaf bytes are truly canonical deterministic JSON
- `canonical_json_hash` matches the SHA-256 of those canonical leaf bytes
- `worm_ed25519_sig` verifies over the ciphertext-based evidence digest
- the leaf payload contains actor attribution, including `source_ip`
- every Merkle proof step is correctly encoded and the full RFC6962 proof reconstructs the supplied root
- the chained daily-root signature verifies against `SHA256(prev_root_hash || merkle_root)`

On success the verifier prints `ACCEPT <timestamp>`. On failure it exits after printing a specific rejection reason. There is no permissive mode, partial validation mode, or unsigned export path in the released tool.

### 4.4 CLI Verification Guarantees

Two command-line verification surfaces ship in the preserved baseline.

1. `tools/verify_forensics.go`
   - Scope: offline forensic bundle verification
   - Trust root: `/etc/ransomeye/worm_signing.pub`
   - Guarantee: rejects any bundle with invalid bundle signature, invalid WORM signature, invalid canonical JSON, invalid inclusion proof, invalid chained root signature, or missing actor attribution

2. `cmd/ransomeye-cli/main.go` via `ransomeye validate --full`
   - Scope: release-operational validation
   - Guarantee: fails on PRD hash mismatch, missing or unloadable TLS key pairs, invalid WORM key permissions or size, missing `POSTGRES_DSN`, failed database ping, migration-set mismatch, missing Merkle root, or invalid nginx configuration

The first CLI is forensic and offline. The second CLI is operational and deployment-oriented. They are complementary, not interchangeable.

## 5. Validation Coverage

### 5.1 `scripts/full_system_validation.sh`

The released final validation script is strict fail-fast (`set -euo pipefail`) and stops at the first failed control. It emits only `[PASS]` and `[FAIL]` lines and exits non-zero on any failure.

The shipped checks are:

1. `pgrep -x ransomeye-core`
   - Core process must be running

2. Listener policy check via `ss`
   - Ports `50052`, `50053`, `5432`, `6379`, and `8443` must all be present on `127.0.0.1`
   - None of those ports may appear on `0.0.0.0`
   - None of those ports may appear on `::1`

3. TLS validation against `127.0.0.1:8443`
   - CA file `/opt/ransomeye/core/certs/ca.crt` must exist
   - TLS 1.3 probe uses `openssl s_client -tls1_3 -verify_return_error -CAfile /opt/ransomeye/core/certs/ca.crt`
   - Output must contain `Protocol: TLSv1.3`
   - Output must contain `Verify return code: 0 (ok)`
   - TLS 1.2 probe must not negotiate
   - TLS 1.2 probe output must contain an explicit failure indicator such as `alert protocol version`, `handshake failure`, `unsupported protocol`, `wrong version number`, or `no protocols available`

4. WORM key validation
   - `/etc/ransomeye/worm_signing.key` must exist
   - file mode must be `0600`
   - file size must be exactly 32 or 64 bytes

5. Merkle presence validation
   - `POSTGRES_DSN` must be set
   - `SELECT COUNT(*) FROM merkle_daily_roots` must return a positive integer

6. PRD integrity validation
   - `cd prd && sha256sum *.md | sort | diff prd.sha256 -`
   - any manifest drift fails the script

7. nginx validation
   - `nginx -t` must succeed

8. UI validation
   - `curl -ksS https://127.0.0.1/` must return HTML

9. API health validation
   - `curl -ksS https://127.0.0.1/api/v1/health` must return JSON containing `"status":"ok"`

### 5.2 PRD Integrity Validation

PRD integrity is enforced in three separate locations in the preserved baseline.

- `scripts/full_system_validation.sh`
  - compares the manifest directly with recomputed sorted file hashes

- `.git/hooks/pre-commit`
  - blocks any commit where PRD markdown changes without a synchronized `prd/prd.sha256`
  - blocks any direct `prd.sha256` modification without corresponding PRD markdown changes
  - recomputes the prepared manifest and rejects mismatches before commit completion

- `cmd/ransomeye-cli/main.go`
  - parses `prd/prd.sha256`
  - hashes every `prd/*.md` file individually
  - rejects duplicate manifest entries, malformed lines, missing files, and hash mismatches

The release line therefore checks PRD integrity both at authoring time and at runtime validation time.

### 5.3 TLS Validation Behavior

The operational validation behavior for the release line is fail-closed.

- Missing CA file is fatal
- Untrusted certificate chain is fatal
- Negotiation of any protocol other than TLS 1.3 on the validated API surface is fatal
- A TLS 1.2 handshake that succeeds is fatal
- A TLS 1.2 handshake that does not succeed but also does not emit an explicit protocol failure string is still fatal

Outside the shell validator, the runtime database trust gate independently rejects:

- missing server certificate
- non-TLS transport
- incomplete handshakes
- non-TLS1.3 connections
- missing peer certificate
- mismatched PostgreSQL server leaf fingerprint
- unexpected SAN contents
- non-loopback remote address
- client certificate CN / database role mismatch

### 5.4 Database Integrity Checks

Database integrity is validated at three depths in the released baseline.

1. `scripts/full_system_validation.sh`
   - confirms the presence of at least one `merkle_daily_roots` row

2. `cmd/ransomeye-cli/main.go`
   - verifies that `schema_migrations` exactly matches the SQL files under `migrations/`
   - rejects missing migration versions
   - requires at least one valid 64-hex-character Merkle root row

3. `core/internal/db/validator/validation_runner.go`
   - verifies required tables exist
   - verifies forbidden tables do not exist
   - verifies immutability enforcement by attempting forbidden updates and deletes
   - verifies Merkle chain rows and recomputed root values
   - verifies row-level security isolation
   - verifies Timescale hypertables
   - verifies required indexes
   - reports pass/fail as structured JSON

The released validator expects 42 migrations in the schema set. That count is part of the v0.0 baseline and any drift is a validation failure.

## 6. Known Constraints

### 6.1 Single-Node Architecture

The preserved v0.0 baseline is a single-node deployment model.

- Core API, nginx, PostgreSQL, Redis, AI, and SINE are assumed to reside on the same host
- The final validation script only validates one local host
- No distributed consensus, multi-node replication, or multi-writer forensic quorum exists in this release
- Merkle daily roots are chained within the single-node database, not across independent witnesses

### 6.2 No HSM or TPM-Backed Key Custody Yet

The released baseline does not bind private key custody to hardware.

- `/etc/ransomeye/worm_signing.key` is a filesystem-backed Ed25519 private key
- `/etc/ransomeye/pki/intermediate_ca.key` is a filesystem-backed Ed25519 signing key
- `/opt/ransomeye/postgres/certs/server.key` is a filesystem-backed PostgreSQL server private key
- Access control is enforced through file mode restrictions, not HSM-backed non-exportability
- TPM attestation is not an active cryptographic trust root in the released implementation; V0.0 documentation still treats TPM quoting as stubbed

This means compromise of host root access remains sufficient to access local signing keys in the preserved v0.0 model.

### 6.3 CPU-First Inference

The preserved release baseline is CPU-first by design.

- PRD baseline requires inference to function without GPU
- Local AI and offline inference paths assume CPU execution is the minimum viable platform
- SIMD and BLAS acceleration are opportunistic performance optimizations, not trust anchors
- GPU absence is not a release failure condition

This matters for preservation because performance tuning and acceleration choices are subordinate to determinism and offline operability in v0.0.

### 6.4 Air-Gap Dependency Model

The preserved baseline assumes an air-gapped deployment model.

- Runtime trust roots are provisioned locally
- Package registries, cloud KMS, SaaS certificate validation, external telemetry brokers, and online model fetches are not part of the release dependency set
- Compliance code treats non-loopback DNS resolvers and a default route as air-gap violations
- Signed bundles, local certificates, prepared manifests, and local configuration files are the only release-time trust inputs

Operationally, this means a valid v0.0 deployment must be self-sufficient after provisioning. If continued operation depends on internet reachability, that deployment is outside the preserved release baseline.

### 6.5 Released AI Loopback Transport Asymmetry

The preserved v0.0 codebase contains a transport asymmetry on the AI loopback path.

- The AI service process requires a secure gRPC port with client authentication
- The core-side AI client still uses insecure gRPC credentials for the loopback dial
- `scripts/full_system_validation.sh` does not test the AI gRPC transport directly

This document records that asymmetry as part of the frozen baseline so later preservation work does not accidentally rewrite release history by assuming end-to-end AI loopback mTLS was uniformly enforced in v0.0.
